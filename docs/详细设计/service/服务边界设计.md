# TradingClaw 后端 Service 详细设计：服务边界

## 1. 设计目标

- 明确各服务的职责边界、数据主权与调用关系。
- 保证 MVP 可快速交付，同时支持 V1/V2 服务化拆分。
- 统一跨服务协作模式，减少跨域耦合和口径不一致问题。

## 2. 领域与服务映射

| 领域 | 服务名 | 对应需求ID | 版本定位 |
| --- | --- | --- | --- |
| 应用入口 | app-service | APP-001 | MVP |
| 用户与会话 | identity-service | USR-001~003 | MVP |
| 资产行情 | market-service | MKT-001~004 | MVP/V1 |
| 证券交易 | securities-trade-service | SEC-001~005 | MVP/V1/V2 |
| 策略引擎 | strategy-service | STR-001~004 | MVP/V1/V2 |
| 统一交易抽象 | unified-trade-kernel | UTL-001 | MVP |
| 交易计划 | planning-service | PLN-001~006 | V1/V2 |
| 智能能力 | intelligence-service | AI-001/002、OCR-001 | V1 |

## 3. 服务职责与边界

## 3.1 app-service

- 边界职责：统一入口、鉴权链路挂载、限流、幂等校验、统一响应、错误码转换。
- 不负责：业务规则计算、策略状态迁移、交易通道调用。
- 依赖方向：仅调用应用服务接口，不直接访问领域数据库。

## 3.2 identity-service

- 边界职责：注册登录、会话签发与失效、用户资料查询、额度与可用积分、账户归属关系管理。
- 不负责：行情查询、下单执行、策略计算。
- 边界约束：任何交易类与策略类写操作必须先通过身份与归属校验。

## 3.3 market-service

- 边界职责：证券/基金实时行情、历史行情、指标计算、历史缺口补齐、缓存回退。
- 不负责：交易下单、策略持仓状态决策。
- 边界约束：对外统一返回数据时间戳、数据状态（fresh/stale）与来源标识。

## 3.4 securities-trade-service

- 边界职责：券商登录、会话状态、资产持仓、下单撤单、委托成交、申购能力。
- 不负责：策略额度判断、交易计划评审发布。
- 边界约束：必须通过统一交易抽象暴露能力，不直接给策略层开放 SDK。

## 3.5 strategy-service

- 边界职责：策略创建、修改、停启、下线、归档；策略计算与调仓建议生成。
- 不负责：账户注册登录、券商底层协议细节。
- 边界约束：状态机迁移必须先写状态日志再落状态变更。

## 3.6 unified-trade-kernel

- 边界职责：统一下单/撤单/查单/持仓/成交模型；通道适配器注册和路由。
- 不负责：具体通道业务策略、投资问答、计划发布。
- 边界约束：策略与计划相关执行动作必须通过统一交易模型发起。

## 3.7 planning-service

- 边界职责：计划身份管理、版本管理、评审、校验、发布、执行记录、复盘聚合。
- 不负责：券商协议调用、模型推理执行。
- 边界约束：同交易日同标的计划身份唯一；发布只允许基于当前内容版本。

## 3.8 intelligence-service

- 边界职责：普通问答、投资问答、模型配置读写、OCR 验证码识别。
- 不负责：真实交易执行与资金侧变更。
- 边界约束：问答工具调用失败不得影响核心交易链路可用性。

## 4. 数据主权划分

| 数据对象 | 主权服务 | 只读消费者 | 说明 |
| --- | --- | --- | --- |
| 用户、会话、账户绑定 | identity-service | app/strategy/planning | 交易前置校验基线 |
| 行情缓存、历史补齐任务 | market-service | strategy/planning/intelligence | 提供状态可见性 |
| 券商会话、委托、成交回执 | securities-trade-service | unified-trade-kernel/strategy/planning | 不跨服务直接写入 |
| 策略实体与状态日志 | strategy-service | app/planning | 策略生命周期唯一事实源 |
| 统一交易契约与适配器元数据 | unified-trade-kernel | strategy/planning | 平台级能力元数据 |
| 计划、版本、评审、校验、发布、执行 | planning-service | app/strategy/intelligence | 计划链路唯一事实源 |
| 模型配置、问答上下文、OCR 记录 | intelligence-service | app | 仅提供受控查询接口 |

## 5. 跨服务调用规则

- 规则一：调用必须走显式应用接口，不允许跨服务直连数据库。
- 规则二：强一致写链路采用同步调用，副作用采用事件异步处理。
- 规则三：跨服务传输对象使用稳定契约对象，禁止泄露内部实体结构。
- 规则四：任一服务失败必须返回标准错误码与可读失败原因。
- 规则五：交易链路调用必须携带请求ID、用户标识、会话标识与幂等键。

## 6. 对外接口分组建议

## 6.1 公共入口分组

- /api/v1/auth：注册、登录、登出、会话校验。
- /api/v1/user：资料、额度、积分、绑定账户。
- /api/v1/market：实时、历史、指标。
- /api/v1/trade：登录状态、资产、持仓、下单、撤单、流水。
- /api/v1/strategy：策略生命周期与计算。
- /api/v1/plans：计划、版本、评审、发布、校验、复盘。
- /api/v1/ai：问答、模型配置、模型列表、OCR。

## 6.2 内部能力接口

- /internal/trade-kernel/*：统一交易抽象内部接口。
- /internal/events/*：事件订阅与回放接口。
- /internal/health/*：健康检查与依赖状态探测。

## 7. 服务化拆分触发条件

- 当单服务研发变更冲突持续升高且影响主线交付时拆分。
- 当单域吞吐与资源模型明显异构时拆分。
- 当单域故障隔离要求提升并影响核心 SLA 时拆分。
- 当跨团队并行需求显著增加时拆分。

## 8. 验收检查项

- 每个服务均有明确职责、不重叠、不缺失。
- 每个数据对象均存在唯一主权服务。
- 每条核心链路有明确同步调用路径。
- 每个失败场景可定位到具体服务边界并返回可读错误。
