# TradingClaw 后端 Service 详细设计：事件模型

## 1. 设计目标

- 为跨服务协作提供统一事件语义，降低同步链路耦合度。
- 保障交易、策略、计划等核心流程的可追踪与可恢复能力。
- 通过可回放事件支撑质量分析、复盘与审计。

## 2. 事件总线与投递策略

## 2.1 总线角色

- 事件发布方：领域服务在本地事务提交后发布领域事件。
- 事件总线：接收、持久化、分发、重试、死信管理。
- 事件订阅方：按订阅规则处理事件并更新本域读模型。

## 2.2 投递保障

- 发布保障：Outbox 事务消息，确保“状态变更”和“事件发布”同源一致。
- 消费保障：至少一次投递，消费端以幂等键去重。
- 异常保障：重试达到阈值进入死信队列并触发告警。

## 3. 事件统一结构

| 字段 | 说明 | 必填 |
| --- | --- | --- |
| event_id | 事件唯一ID | 是 |
| event_type | 事件类型 | 是 |
| event_version | 事件版本 | 是 |
| occurred_at | 事件发生时间 | 是 |
| producer | 发布服务 | 是 |
| trace_id | 请求追踪ID | 是 |
| subject_id | 主体标识（用户/策略/计划） | 是 |
| aggregate_id | 聚合根标识 | 是 |
| idempotency_key | 幂等键 | 是 |
| payload | 事件载荷 | 是 |
| metadata | 附加元信息 | 否 |

## 4. 核心事件清单

## 4.1 用户与会话事件

- user.registered：用户注册成功。
- session.created：会话创建成功。
- session.invalidated：会话失效或登出。
- account.binding.changed：账户绑定关系变更。

## 4.2 行情事件

- market.snapshot.updated：实时快照更新。
- market.history.backfilled：历史缺口补齐完成。
- market.data.degraded：外部源失败进入缓存回退。

## 4.3 交易事件

- trade.order.submitted：订单提交成功受理。
- trade.order.rejected：订单被拒绝并返回可读原因。
- trade.order.filled：订单成交状态变化。
- trade.order.cancelled：订单撤销成功。
- trade.session.state.changed：交易会话状态变化。

## 4.4 策略事件

- strategy.created：策略创建成功。
- strategy.state.changed：策略状态迁移。
- strategy.rebalanced：调仓建议生成。
- strategy.archived：策略归档完成。

## 4.5 交易计划事件

- plan.created：计划创建成功。
- plan.version.created：新版本创建。
- plan.review.completed：评审完成并产生结论。
- plan.validation.completed：校验完成并产出分级问题。
- plan.published：发布成功。
- plan.execution.recorded：执行订单记录写入。

## 4.6 智能能力事件

- ai.answer.generated：问答结果生成。
- ai.model.config.changed：模型配置变更。
- ocr.captcha.recognized：OCR 识别完成。

## 5. 主题与分区规划

| 主题 | 事件范围 | 分区键 |
| --- | --- | --- |
| topic.identity | 用户与会话 | user_id |
| topic.market | 行情与补齐 | symbol |
| topic.trade | 交易与订单 | account_id |
| topic.strategy | 策略生命周期 | strategy_id |
| topic.plan | 交易计划 | plan_id |
| topic.ai | AI 与 OCR | user_id |

规划原则：

- 同一聚合根事件必须路由到同一分区保证顺序消费。
- 不同聚合根允许并行消费以提升吞吐。

## 6. 消费模型设计

## 6.1 同步链路后置消费

- 下单成功后：异步写审计日志、统计指标、复盘数据。
- 策略状态变化后：异步更新看板与提醒消息。
- 计划发布后：异步触发质量归档与历史快照。

## 6.2 消费幂等规范

- 幂等键建议：event_id 或 business_id + event_type + version。
- 幂等存储：按消费者维度保存消费指纹。
- 重放策略：支持按时间窗口与事件类型回放。

## 7. 失败处理与补偿

- 可重试错误：网络超时、下游临时不可用，按指数退避重试。
- 不可重试错误：参数非法、状态冲突，直接写死信并记录原因。
- 补偿动作：由补偿任务依据死信记录执行人工或自动补偿。

## 8. 可观测与审计

- 指标：发布成功率、消费延迟、重试次数、死信积压量。
- 日志：每条事件记录 trace_id、event_id、consumer、处理结果。
- 审计：关键事件保留可检索轨迹，支持按主体和时间回溯。

## 9. 版本演进策略

- 事件契约采用向后兼容扩展，新增字段仅追加不删除。
- event_version 采用整数递增，消费者按版本做兼容解析。
- 废弃事件必须保留一段灰度期，完成消费者迁移后下线。

## 10. 验收检查项

- 核心状态变更均有对应事件发布。
- 事件发布与业务落库具备一致性保障。
- 订阅失败可自动重试并可观测。
- 死信可回放、可定位、可补偿。
