# TradingClaw 后端 Service 详细设计：部署拓扑

## 1. 设计目标

- 支撑 MVP 核心链路稳定上线并具备故障隔离能力。
- 满足缓存回退、事件驱动、统一鉴权和可观测要求。
- 为 V1/V2 的服务拆分与容量扩展提供标准部署基线。

## 2. 环境分层

| 环境 | 目标 | 特征 |
| --- | --- | --- |
| dev | 研发联调 | 单实例优先、快速迭代、低成本 |
| test | 测试验收 | 接近生产拓扑、可回归、可压测 |
| prod | 生产运行 | 多副本高可用、可灰度、可回滚 |

## 3. MVP 部署拓扑

## 3.1 组件清单

- gateway：统一入口层，承载鉴权、限流、幂等、响应标准化。
- app-monolith：模块化单体进程，承载 identity/market/securities/strategy/trade-kernel。
- db-main：关系型数据库，存储核心业务事实数据。
- redis-cache：缓存层，承载高频查询与降级回退。
- mq-bus：消息总线，承载领域事件与异步任务。
- obs-stack：日志、指标、链路追踪。

## 3.2 请求路径

- 外部请求 -> gateway -> app-monolith -> db/redis/mq -> 响应。
- 交易执行请求必须经过 identity 校验和 trade-kernel 路由。
- 异步任务通过 mq-bus 触发统计、审计、复盘更新。

## 4. V1/V2 演进拓扑

## 4.1 V1 拆分建议

- 从 app-monolith 拆分 planning-service 与 intelligence-service。
- gateway 增加按路由转发策略：
  - /api/v1/plans/* -> planning-service
  - /api/v1/ai/* -> intelligence-service
- 保持 trade 相关核心链路仍在主服务，降低交易风险。

## 4.2 V2 拆分建议

- 拆分 crypto-trade-service，承接数字资产通道能力。
- trade-kernel 独立部署为平台能力层，统一证券和数字资产适配。
- 引入多集群或分区部署，隔离高波动交易通道风险。

## 5. 网络与安全域

## 5.1 网络分区

- dmz 区：仅 gateway 暴露公网入口。
- service 区：业务服务互访，仅开放必要端口。
- data 区：数据库、缓存、消息总线，仅 service 区可访问。

## 5.2 访问控制

- 服务间访问使用服务身份认证与最小权限策略。
- 管理接口与内部接口分离，内部接口仅内网可达。
- 敏感配置统一通过密钥管理系统注入。

## 6. 高可用与容灾

## 6.1 高可用策略

- gateway 至少双实例部署。
- 核心业务服务按无状态多副本部署。
- db-main 主从或主备部署，支持自动故障切换。
- redis-cache 与 mq-bus 使用高可用部署模式。

## 6.2 容灾策略

- 关键数据定期备份与异地备份。
- 提供 RPO/RTO 目标并定期演练恢复。
- 外部依赖故障时启用缓存回退并标识数据状态。

## 7. 扩缩容策略

- gateway 按 QPS 和 P95 延迟触发扩容。
- app 或拆分服务按 CPU、内存、请求排队长度自动扩容。
- mq 消费者按分区滞后量和消费延迟扩容。
- redis 与 db 通过容量阈值和热点检测触发纵向或横向扩展。

## 8. 发布与回滚策略

## 8.1 发布策略

- 采用蓝绿或灰度发布，逐步放量。
- 核心交易变更需先影子流量验证再全量。
- 发布前执行健康检查、依赖检查、迁移脚本校验。

## 8.2 回滚策略

- 应用版本可快速回滚到上一稳定版本。
- 数据变更采用可回滚脚本或前向兼容迁移。
- 事件契约变更采用双写或兼容窗口避免回滚中断。

## 9. 可观测拓扑

- 日志：集中采集 gateway 与所有服务日志，支持 trace_id 检索。
- 指标：采集服务 SLA、缓存命中率、消息延迟、错误码分布。
- 追踪：覆盖 gateway -> service -> adapter -> 外部依赖全链路。
- 告警：按核心链路失败率、依赖可用率、死信积压设置分级告警。

## 10. 容量基线建议

- MVP 初期以 2x 冗余为最小可用配置。
- 交易时段按峰值流量预留 30% 以上余量。
- 行情与问答场景分别设独立限流，避免互相抢占资源。

## 11. 验收检查项

- 核心服务具备多副本可用能力。
- 外部依赖失败可降级且不阻断核心链路。
- 发布、回滚、容灾流程可演练并通过。
- 可观测系统可在 30 分钟内定位核心故障来源。
