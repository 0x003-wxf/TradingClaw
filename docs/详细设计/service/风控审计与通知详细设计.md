- 本文档覆盖 `audit-risk-service` 与 `notification-service`。
- 目标是形成风险前置裁决、旁路审计、人工复核、通知闭环。
- 本模块为交易、策略、AI 等模块提供统一治理能力。

## 2. 模块职责

### 2.1 `audit-risk-service`

- 风险规则匹配与裁决。
- 审计日志归集。
- 风险事件记录、人工复核、闭环处置。

### 2.2 `notification-service`

- 系统通知、策略通知、风险通知、异常告警发送。
- 通知投递结果回写和失败重试。

## 3. 领域边界

- 风控模块提供裁决，不直接修改交易或策略主状态。
- 审计模块记录关键动作，但不成为业务事实源。
- 通知模块消费事件并投递，不反向持有业务主状态。

## 4. 核心模型

| 对象 | 说明 |
| --- | --- |
| `AuditLog` | 审计记录 |
| `RiskRule` | 风险规则 |
| `RiskEvent` | 风险事件 |
| `ManualReview` | 人工复核记录 |
| `Notification` | 通知记录 |
| `DeliveryAttempt` | 投递尝试记录 |

## 5. 核心流程

### 5.1 交易前风控裁决

1. 接收交易网关或策略运行时的风险校验请求。
2. 校验身份、账户归属、额度、状态、参数越界和风险规则。
3. 返回通过、拒绝或需要人工复核。
4. 对命中规则的请求记录风险事件。

### 5.2 审计旁路消费

1. 消费登录、下单、撤单、策略启动、AI 工具调用等关键事件。
2. 写入审计日志。
3. 关联 `trace_id`、`request_id`、业务主键。

### 5.3 人工复核闭环

1. 高风险事件进入 `PENDING_REVIEW`。
2. 人工或自动复核形成结论。
3. 更新到 `CONFIRMED`、`DISMISSED` 或 `MITIGATED`。
4. 最终进入 `CLOSED`。

### 5.4 通知投递

1. 接收 `notification.requested` 事件。
2. 选择通知渠道和模板。
3. 执行投递并记录结果。
4. 发布 `notification.delivered` 或失败重试记录。

## 6. 状态机

### 6.1 风险事件状态机

- `DETECTED`
- `PENDING_REVIEW`
- `CONFIRMED`
- `DISMISSED`
- `MITIGATED`
- `CLOSED`

规则：

- 命中高风险规则必须进入 `DETECTED`。
- 需要人工介入时进入 `PENDING_REVIEW`。
- 只有处置完成后才能进入 `MITIGATED` 再闭环。

## 10. 实施要点

- 风控裁决必须在交易和策略执行前接入，不允许前端或 CLI 替代。
- 审计日志禁止静默删除。
- 通知模块必须支持失败重试、幂等和结果回写。
