# TradingClaw 后端 Service 架构方案

## 1. 目标与适用范围

本方案用于承接后端需求文档中的服务化落地设计，覆盖以下目标：

- 在 MVP 阶段优先保障 P0 链路可交付、可验证、可追踪。
- 通过统一领域边界和统一交易抽象，降低多通道接入复杂度。
- 支持从 MVP 向 V1/V2 平滑演进，避免一次性微服务化带来的交付风险。

适用范围：USR、MKT、SEC、STR、UTL、APP 为 MVP 核心；CRY、NEWS、AI、OCR、PLN 为 V1/V2 扩展重点。

## 2. 总体架构选择

采用“模块化单体 + 领域分层 + 事件驱动增强”的混合架构。

### 2.1 架构原则

- 统一入口：所有请求从 API 入口进入，统一做鉴权、限流、幂等、审计。
- 领域隔离：按业务域划分模块，禁止跨域直接访问数据表。
- 同步与异步分流：核心交易链路走同步，统计聚合与副作用走异步。
- 适配器解耦：外部行情、证券、数字资产、模型能力全部经适配器访问。
- 可演进拆分：模块边界即未来服务边界，先模块化再服务化。

### 2.2 分层模型

- 接入层：API Gateway、统一错误码映射、响应清洗。
- 应用层：用例编排、事务边界控制、权限校验串联。
- 领域层：实体、值对象、领域服务、状态机、领域事件。
- 基础设施层：Repository、缓存、消息总线、第三方通道适配器。

## 3. 服务边界与职责

## 3.1 app-service（APP-001）

- 职责：统一启动入口、模块注册、日志与追踪初始化、响应标准化。
- 输入：全量外部请求。
- 输出：标准响应结构、统一错误码、追踪上下文。

## 3.2 identity-service（USR-001~003）

- 职责：注册登录、会话管理、用户资料、策略额度和积分可用值、账户归属关系校验。
- 关键规则：交易前必须校验会话有效性与账户归属一致性。
- 数据主权：用户、会话、账户绑定关系、积分冻结与释放记录。

## 3.3 market-service（MKT-001~004）

- 职责：实时行情、历史行情、指标计算、缺口补齐与缓存回退。
- 关键规则：外部行情失败时优先返回可用缓存并标记数据状态。
- 数据主权：行情缓存、历史补齐任务状态、指标计算快照。

## 3.4 securities-trade-service（SEC-001~005）

- 职责：证券登录、会话状态、资产持仓、下单撤单、交易流水、申购能力。
- 关键规则：登录失败必须返回可读原因；下单撤单必须回执业务状态。
- 数据主权：交易会话镜像、订单回执、成交与委托快照。

## 3.5 strategy-service（STR-001~004）

- 职责：策略生命周期、策略计算能力、场内/场外策略统一管理。
- 关键规则：策略状态机严格遵循“待启动/运行中/已停止/已下线/已归档”。
- 数据主权：策略定义、策略状态流转日志、策略计算结果。

## 3.6 unified-trade-kernel（UTL-001）

- 职责：统一交易模型、统一下单撤单查单持仓成交接口、适配器注册中心。
- 关键规则：策略层仅依赖统一抽象，不允许直接调用具体通道 SDK。
- 数据主权：交易抽象契约、通道适配能力元数据。

## 3.7 planning-service（PLN-001~006，V1重点）

- 职责：计划创建、版本管理、评审发布、校验质量、执行记录与复盘聚合。
- 关键规则：同日同标的计划身份唯一；发布前校验与评审必须命中当前内容版本。
- 数据主权：计划主表、版本表、评审表、校验结果表、执行与快照表。

## 3.8 intelligence-service（AI-001/002，OCR-001，V1重点）

- 职责：普通/投资问答、模型配置持久化、模型列表读取、OCR 验证码识别。
- 关键规则：工具调用可扩展但不得影响主流程稳定性。
- 数据主权：模型配置、问答上下文索引、OCR 请求与结果日志。

## 4. 服务交互设计

## 4.1 同步链路

- 登录与鉴权链路：app-service -> identity-service。
- 下单链路：app-service -> identity-service（主体校验）-> strategy-service（额度与规则）-> unified-trade-kernel -> securities-trade-service 或 crypto-adapter。
- 计划发布链路：app-service -> planning-service（发布可用性检查）-> 发布提交。

## 4.2 异步链路

- 领域事件总线：策略状态变更、计划发布、订单回执、校验完成统一发布领域事件。
- 订阅处理：统计、质量看板、复盘聚合、审计归档通过事件消费更新。

## 4.3 失败与降级策略

- 外部依赖失败：优先返回缓存副本并标注 stale 状态与缓存时间。
- 通道不可用：返回标准错误码、可读原因、可重试建议。
- 异步积压：限流与重试分级，超阈值触发告警与保护性降级。

## 5. 数据与一致性方案

## 5.1 存储策略

- 核心交易与状态机数据使用关系型数据库，保证事务一致性。
- 高频查询结果使用缓存层，提高响应性能并削峰。
- 审计与观察数据写入日志存储或时序检索系统。

## 5.2 一致性策略

- 单域内：本地事务保证强一致。
- 跨域：Outbox + 事件投递保证最终一致。
- 幂等：订单写入、执行记录、状态迁移均要求业务幂等键。

## 6. 安全与可观测设计

## 6.1 安全控制

- 统一鉴权中间件：会话校验、令牌校验、主体一致性校验全链路开启。
- 输入防护：参数白名单、注入拦截、越权检查。
- 敏感信息处理：日志与响应字段统一脱敏，不输出密钥/令牌/密码明文。

## 6.2 可观测控制

- 统一 TraceId：贯穿入口、领域事件、外部调用。
- 结构化日志：至少包含请求ID、用户标识、模块标识、错误码。
- 指标体系：缓存命中率、核心链路成功率、外部依赖可用率、状态机非法迁移次数。

## 7. 版本演进与拆分路线

## 7.1 MVP（P0优先）

- 形态：单仓模块化单体。
- 优先交付：APP、USR、MKT、SEC、STR、UTL。
- 目标：先打通登录、行情、下单/撤单、策略操作主链路。

## 7.2 V1（能力扩展）

- 扩展：PLN、NEWS、AI、OCR、CRY 基础能力。
- 拆分建议：优先拆 planning-service 与 intelligence-service，降低对交易主链路扰动。

## 7.3 V2（高阶能力）

- 扩展：策略重建收益测算、数字资产深度接入、执行复盘增强。
- 拆分建议：将 crypto 相关能力独立为服务，形成多通道交易中台能力。

## 8. 实施清单

- 先完成统一错误码、统一响应结构、统一鉴权中间件。
- 建立 unified-trade-kernel 契约与适配器注册机制。
- 建立策略与计划两套状态机执行器与状态日志。
- 建立缓存回退标准结构和数据状态标记规范。
- 建立 Outbox 事件框架和核心事件主题定义。
- 建立核心链路压测与故障演练基线。
